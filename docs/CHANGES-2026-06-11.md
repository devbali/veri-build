# Veri Build & Veri DSL Changes — 2026-06-11

## Implemented Changes (since last commit)

Ordered by significance.

### 1. Backend registry system (`src/veri_build/backend/`)

**Why:** The pipeline had hardcoded targets (`if target == 'fstar'`) throughout
the runner, pipeline.py, and lint — making it impossible to add new output
languages without touching every file.

**What:**
- Created `src/veri_build/backend/` with a plugin-style registry:
  - `base.py` — abstract `Backend` class with `agent_extra_rules()`,
    `verify_extraction()`, `dsl_language()`, `self_check_command()`
  - `fstar_c.py` — F* → C via KaRaMeL (owns the Low* rules: no `*`, no inner
    let rec, no `seq Prims.bool`)
  - `fstar_ocaml.py` — F* → OCaml via `fstar --codegen OCaml` (zero extra rules)
  - `fstar_wasm.py` — F* → Wasm via krml `-wasm` (same Low* rules as C)
  - `dafny_java.py` — Dafny → Java via `dafny translate java`
  - `dafny_js.py` — Dafny → JavaScript via `dafny translate js`
  - `dafny_python.py` — Dafny → Python via `dafny translate py`
- `Backend.dsl_language()` returns `'fstar'`, `'dafny'`, or `'python'` to
  select the right DSL printer/parser — the rest of the pipeline uses this
  instead of hardcoded `target == 'fstar'` checks.
- `docs/BACKENDS.md` documents how to add a new backend.

**Files:** `src/veri_build/backend/*.py` (8 new files), `docs/BACKENDS.md` (new)

### 2. Agent prompt decoupled from krml-specific rules

**Why:** The agent prompt had krml rules (no `*`, no inner `let rec`, etc.)
hardcoded. The OCaml and Dafny backends don't need these restrictions.

**What:** `launch_agent()` now calls `backend.agent_extra_rules()` for
backend-specific rules and `backend.self_check_command()` for the extraction
self-check step. The generic prompt handles the "copy val signatures verbatim,"
"no Pure keywords," "all functions required" rules that apply to all backends.

**Files:** `scripts/veri-build-runner` (agent prompt section)

### 3. `_resolve_imports()` in `spec.py` — cross-module type inlining

**Why:** `import DeadlineModel` in a Veri DSL block did nothing at the parser
level. Each block was parsed independently, so type references across specs
were unresolved.

**What:** `_resolve_imports()` finds `X.veri.md` (or `x.veri.md` via
CamelCase→snake_case), extracts only type declarations (class, type,
CONSTRAINT), and inlines them into the importing block.

**Files:** `src/veri_build/spec.py`

### 4. Agent rewrite — multi-turn with self-check

**Why:** The old agent got a vague prompt and had no way to check its own
output. The parent couldn't verify correctness.

**What:**
- Agent gets the F* interface (correct type names) instead of raw Veri DSL
- Multi-turn: `--allowedTools Bash,Read,Write --dangerously-skip-permissions`
- Self-check required: write .fst, run fstar, then run krml
- Parent re-verifies by injecting code into the generated interface
- 3-round retry loop with function-count feedback
- Env var passthrough for `ANTHROPIC_*`
- Code post-processing: strip fences, type fixes, Pure effect stripping

**Files:** `scripts/veri-build-runner` (launch_agent, _run_agent)

### 5. `_check_functions_implemented()` + lint enforcement

**Why:** Lint passed specs with contracts but no bodies. These were
unimplemented interfaces that looked "verified."

**What:** Every `def` with `REQUIRES`/`ENSURES` must have a body (`return`,
`if`, or `match`), `EXTERN`, or `#TODO`. Lint rejects contract-only functions.

**Files:** `src/veri_build/pipeline.py`

### 6. DSL parser fixes (submodule)

**Why:** Several Veri DSL constructs failed to parse or were silently dropped.

**What (in `src/veri_build/dsl/src/`):**

| Fix | File | Issue |
|---|---|---|
| Added `elif` keyword | `veri_parser.py` | `elif` was tokenized as IDENT, broke if/elif/else |
| Leading if-form in expression parser | `veri_parser.py` | Only handled ternary `A if B else C`, not `if B: A else: C` |
| Body consume for non-return expressions | `veri_parser.py` | Functions with body starting with `if` had body silently dropped from token stream |
| REQUIRES expression stops at comparison | `veri_parser.py` | `_parse_expr()` greedily consumed trailing `if` as ternary |
| ValDecl.body field | `veri_ast.py` | Functions with contracts + body lost the body |
| `function method` → `function` | `backend/dafny/printer.py` | Dafny 4 deprecated `function method` |
| Body printing for ValDecl | `backend/dafny/printer.py` | Dafny printer didn't output body when contracts present |
| `for`/`range` keywords removed | `veri_parser.py` | Dead keywords — no AST or printer support |

### 7. Writable `/output` volume mount in `compile_veri`

**Why:** Docker mounts the spec directory as read-only. The runner couldn't
write `.fsti`/`.fst` files anywhere.

**What:** Separate writable `/output` volume mounted at `build/` next to the
spec. Avoids the "can't create mountpoints under read-only parent" Docker
limitation.

**Files:** `src/veri_build/pipeline.py`

### 8. Cross-module F* verification (include_dirs)

**Why:** F* couldn't find depended-on modules during lint (e.g., TauFairScheduler
needing DeadlineModel's `.fst.checked`).

**What:** Added `include_dirs` parameter to `_run_fstar_interface()`. Lint
includes the spec's directory and sibling `build/fstar/`.

**Files:** `src/veri_build/pipeline.py`

### 9. Backend printer import fix

**Why:** Runner used stale imports: `fsti_printer.FStarPrinter` (removed file)
and `dafny_printer.DafnyPrinter` (never existed at that path).

**What:** Changed to `backend.fstar.printer.FStarPrinter` and
`backend.dafny.printer.DafnyPrinter`. Same fix for `convert_to_fcl` parsers.

**Files:** `scripts/veri-build-runner`

### 10. Dockerfile pinned version comments

Added comments explaining why F* v2026.05.31 and Dafny v4.11.0 are pinned,
and what to update if upgrading.

**Files:** `Dockerfile`

### 11. Skill/docs updates

- "Lint before delivering" hard rule
- Every function must have body, EXTERN, or #TODO
- Docker sub-agent permissions guidance
- Sub-agent retry protocol documented

**Files:** `skills/VERIFICATION-SKILL.md`, `docs/VERIFICATION-SKILL.md`

## Proposed Changes (from `VERI_BUILD_CONFUSIONS.md` and `VERI_BUILD_NOTES.md`)

These are recommendations from the debugging process that haven't been
implemented yet.

### 1. Auto-detect LLM model from host environment

**Problem:** The runner hardcodes `deepseek-v4-pro` for OpenClaw and relies
on whatever is configured for Claude. If the host uses a different model, the
runner ignores it.

**Proposal:** Pass through all `ANTHROPIC_*` and `OPENCLAW_*` env vars to
Docker, including the model name. Don't hardcode any model identifiers.

### 2. Exit code handling in `compile_veri`

**Problem:** `compile_veri` ignores Docker exit codes and only reads
`result.json`. If the runner crashes before writing the JSON, the error is
silent.

**Proposal:** Always capture Docker stderr even when result.json is missing.
Report the raw container output as a fallback error message.

### 3. Sequential fst.checked caching for multi-module specs

**Problem:** F* requires `.fst.checked` files for cross-module inlining.
The runner needs to verify depended-on modules before their dependents.
Currently there's no ordering — all modules are verified independently.

**Proposal:** Add a topological sort of specs by their `import` dependencies.
Verify in dependency order. Cache `.fst.checked` files in a shared directory.

### 4. Agent prompt robustness

**Problem:** The agent frequently produces malformed responses:
- Wraps code in markdown fences despite being told not to
- Only implements a subset of TODO functions per response
- Expands compound struct types into flat parameters

**Proposal:** Add stricter response parsing with better error messages for
each failure mode. Consider few-shot examples in the prompt showing the
exact expected CODE response format.

### 5. Veri DSL `match` expression completeness

**Problem:** `match`/`case` is tokenized and parsed but the pattern syntax
is limited. Nested patterns and guards aren't supported.

**Proposal:** Extend `_parse_match` to support or-patterns, guards (`, if cond`),
and nested constructor patterns.

### 6. Runtime type for Dafny backends

**Problem:** Dafny backends produce `.dfy` files but there's no Dafny runtime
library bundled for Java/JS/Python extraction. The user needs the Dafny runtime
to compile the generated code.

**Proposal:** Document the Dafny runtime dependency in BACKENDS.md. Consider
auto-bundling `DafnyRuntime.jar`/`dafny.js`/`dafny.py` in the output directory.
